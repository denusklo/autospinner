## L11 2026-07-07 MaaTouch has no `w` wait — the "device-paced timing" of L10 never existed (and the kill-timer caused stuck touches)
- Symptom: rune occasionally stayed held after a finished spin. Investigation: a script of `w 100`x20 (2s nominal) finished in ~0ms; `w 2000` between down/up executed instantly (process exit 264ms).
- Root cause: MaaTouch v1.1.0 ignores `w` — every "paced" script actually injected contiguously at driver speed (the game accepts this; that's why counters matched exactly). The stuck touch was the p.kill() timer racing MaaTouch's boot: killing the adb session discards unconsumed script including the final `u 0`. Also: stdin EOF kills the remote mid-script (102ms for a 2s script), so streaming+closing is equally unsafe.
- Lesson: next time timing/protocol behavior is assumed, MEASURE it before building on it (a wait that doesn't wait is silent); for run-to-completion semantics over adb, redirect stdin from a device FILE and treat process self-exit as the completion proof — never a kill timer, never stdin EOF.
- Evidence: maat-w-test.js exit 264ms code=137; EOF test exit 102ms; L9/L10's "speed limit" narrative retired.
- Affected files: PROJECT-FACTS P8 rewritten again; phone/autospin.js executeTouchPath rewritten (file-redirect + self-exit + rescue-release).

## L16 2026-07-08 TEMPORAL MINIMUM cracks the "unreadable" bright shock — arcs are additive, so min-over-frames reveals the base
- Symptom: L15 concluded Light shock bases were intrinsically unreadable (refuse + manual override). User pushed: "can we extract the signature? take as many screenshots as you want."
- Root cause of earlier failure: single/few-frame reads see base+arc (additive glare) which flips the apparent hue. The base is invariant; the glare is what varies.
- Lesson: for an ADDITIVE transient overlay (glow, glare, specular flicker), take the per-pixel TEMPORAL MINIMUM across many frames — it strips the overlay and leaves the base, where a few frames' average or consistency-vote cannot. Then subtract the lowest channel to remove any residual white floor and read the pure hue. This turned all 6 shock bases from "refuse" into reliable auto-detection. Frame count matters: min must converge (10 too few → Light residual g/r sinks toward Fire; ~18-20 stable).
- Evidence: temporal-min bases separated all 6 elements cleanly (Wood(22,121,16)…Light(102,61,1)…); Light shocks residual g/r~0.45 vs Fire 0.11; live auto both shocks Light^ weight 91, 2026-07-08.
- Affected files: phone/autospin.js electricBase rewritten (temporal min + hue tree), 18-frame burst; PROJECT-FACTS P11.

## L17 2026-07-08 "Too fast drops the rune" was a stdin-flush bug, not a game speed limit — busy-wait starved libuv so events shipped as one burst
- Symptom: --move-ms 200 (20ms/pt) "untraceable" drop; 210 (21ms/pt) worked. Looked like a sharp game threshold at ~205ms/move.
- Root cause: executeTouchPath busy-waited to each deadline. When per-point lead <= 20ms (i.e. fast speeds) the `if(lead>20) sleep` branch was skipped, so the loop NEVER awaited — libuv never flushed the stdin pipe, Node batched every write, and adb delivered them to the device as a 0.4ms burst (measured via dumpsys RecentQueue). The game saw one instant jump and dropped the rune. At 21ms the sleep branch fired, yielding/flushing per point (device spacing 19ms).
- Lesson: a CPU busy-wait in an async pipeline silently blocks I/O flushing — if you must spin to a deadline, YIELD each spin (`await new Promise(setImmediate)`) so pending writes drain. Verify transport timing at the DESTINATION (dumpsys event ages), not just the sender's write timestamps — they diverged completely (PC 20ms, device 0.4ms).
- Evidence: timing-inspect.js — stepMs=20 device spacing 0.4ms (bug) → 17.9ms (after setImmediate yield); PC-side spacing 20.0ms in both.
- Affected files: phone/autospin.js executeTouchPath busy-wait now yields via setImmediate; PROJECT-FACTS P8.

## L19 2026-07-08 "Bigger beam → fewer combos" was a measurement artifact, not a bug
- Symptom: `--clear-all heart` + both start/end pins returned only 1 combo. A beam sweep looked pathological — 200→5, 800→6, 1600→1 combos ("wider beam does worse", which should be impossible) — screamed bug. Chased it with a scoring change (clearAllMetBonus) that changed nothing.
- Root cause: DoraSolver returns `pick = bestQualified ?? best`. When nothing satisfies the hard demand (clear-all + both pins), it falls back to `best`, which IGNORES the demand. The small-beam "5/6 combo" results were UNMET fallbacks autospin would then ABORT — not valid solutions. Comparing raw comboCount without checking constraint satisfaction compared real clear-all solutions against fallbacks. Measured correctly (firstClearedByType>=total): monotonic — beam 1600→1, 8000→3, 16000→4, all clearing all hearts. No bug; a tight constraint just needs a wide beam.
- Lesson: when A/B-ing solver outputs, ALWAYS report constraint-satisfaction next to the objective — a `bestQualified ?? best` fallback silently violates the demand and out-scores real solutions on the raw metric. One wasted (reverted) code change came from trusting comboCount alone. Confirm the metric measures what you think before theorizing a bug (02 matrix R1-3).
- Evidence: scratchpad clean-evidence.js — beam800 "6c" flagged clearsAllHearts=false (FALLBACK); beam1600/8000/16000 = 1/3/4c all clearsAllHearts=true (1.9s/24s/48s).
- Affected files: none (clearAllMetBonus experiment reverted; algorithm unchanged) — behavioral note only.

## L21 2026-07-08 clear-all CLUMPS runes into one big group (1 combo) — split them with --first-combos N+
- Symptom: `--clear-all dark` (6 darks) cleared all 6 but as ONE 6-rune group = 1 combo, "wasting" a combo (two 3-groups would be 2 combos / more damage). A wider beam (8000, 16000) did NOT split them.
- Root cause: the clear-all steering rewards required runes cleared AND adjacent same-color pairs (pairPotential), so it actively CLUMPS the darks together (more adjacency = higher score); the scoring only mildly prefers 2 groups (+2 weight vs the 6-group's size bonus), not enough to overcome the clumping bias, and the split arrangement is more constrained so the beam doesn't stumble onto it.
- Lesson: to make clear-all produce MULTIPLE combos from the required color, add `--first-combos N+` — it rewards first-wave combo COUNT, which is the "prefer more groups" signal clear-all lacks. Measured: `--clear-all dark --first-combos 5+` split a [6] blob into [3,3] and raised total combos 6→8 (and was faster). This is a flag combination, not a code fix — reach for it before touching solver scoring.
- Evidence: scratchpad dark-firstcombos.js — clear-all only: total6 first3 darks[6]; +first-combos 5+: total8 first6 darks[3,3]; CLI --board dry-run reproduced (8 combos, Dark 6/6 ok, end 5,3).
- Affected files: phone/autospin.js --clear-all help text (usage tip added); no logic change.

## L22 2026-07-08 The constructive clear-all planner DOES work — but only COVERAGE-ONLY (supersedes L20's "don't use the planner" and the earlier "abandoned" note)
- Context: L20 concluded a clear-all MISS is only a beam-width problem and "don't reach for the planner". That holds for a MILDLY scattered scarce type (4 darks → wider beam gathers them). It is FALSE for a badly scattered one: 5 Waters spread to all corners gave 3/5 at beam 8000 AND 16000, no pins — a wider beam never crosses the threshold because 5 can only clear as ONE connected 5-group and the beam won't gather 5 corner runes.
- What actually failed the first two planner attempts: (1) targets were AUGMENTED with extra combo triples → too many cells for routeToTarget to realize + land on the end pin; (2) augmented targets were tried FIRST, starving the routing budget before the working bare-coverage ones; (3) a bug — `this.target` (the combo-count planner's floor) defaults to 5, so solveClearAll silently required firstCombos>=5 and discarded every 1-3-combo clear-all solution as "routing-failed".
- The fix (WORKS): construct COVERAGE-ONLY targets — just the required runes as straight dissolving lines, NO extra combo triples — route many placements, keep the max-combo routed one; add a separate `clearAllComboFloor` (0 unless --first-combos given). Coverage-only targets route reliably WITH the end pin (8/8 water-5-line placements routed to 5,0). autospin now escalates clear-all MISSes to it. Result on the 5-water board: 5/5 cleared + 3 combos + end pin (DoraSolver got 3/5). Slow (~100-150s: it routes every placement to maximize combos) but achieves the goal.
- Lesson: a "construct then route" planner is viable ONLY if the target is MINIMAL (cover the demand, nothing more) — every extra target cell multiplies routing difficulty, especially under an end pin. And before declaring a feature dead, isolate WHY it fails with a direct probe (routeToTarget on a hand-built minimal target succeeded in 2s — that single experiment overturned "the planner can't do it"). L20's beam-first advice is still the fast path for mildly-scattered types; the planner is the fallback the tool now takes automatically.
- Evidence: scratchpad water-gather.js/water-gather-pin.js (bare 5-line routes 5/5 in ~2s, 8/8 placements with end pin), debug-planner.js (found the clearAllComboFloor=5 bug), planner-water.js + CLI dry-run (5/5, 3 combos, end 5,0). verify.js CA1-CA3.

## L23 2026-07-08 Git Bash mangles `/sdcard/...` adb arguments into Windows paths — use PowerShell (or Node execSync) for adb shell commands

- Affected files: none (tooling pitfall only).

- Context: after L61 concluded threading `solveRearrangeConvertAware` was a net regression and reverted the attempt, the User asked "is it not feasible? or impossible?" — a fair challenge to re-examine rather than just repeat the prior conclusion. Re-opened the investigation with fresh controls instead of trusting the earlier (correct-but-incomplete) diagnosis.
- Ran a FAIR control comparison that L61 hadn't actually done: `solveClearAllParallel` at workers=1 (same escalating-beam algorithm, sequential fallback) vs workers=4, holding the algorithm constant — 1950ms → 1003ms, a genuine ~1.9x speedup. This DISPROVED L61's implicit framing ("RearrangeSolver-shaped work just doesn't parallelize well here") — full-effort-per-shard-item concurrency clearly DOES pay off on this machine for comparable heavy `BoardSimulator.resolve()`-based work. Also ran a raw CPU-bound busy-loop across 8 concurrent workers (3038ms solo vs 3047ms at n=8) — perfect scaling, ruling out any OS/virtualization-level throttling.
- With the "maybe this machine just can't do it" theory eliminated, re-examined the FAILED designs' actual mechanics instead of the aggregate timing numbers alone:
  1. Isolated a single warm, no-concurrency worker doing ONE candidate: 7.4s, ~identical to in-process (7.3s) — ruled out spawn/serialization overhead specifically.
  2. Measured PER-CALL timing inside persistent workers, 5 calls each, solo vs 4-way concurrent: solo settled at ~1200-1300ms/call after warmup; 4-way concurrent settled at ~1550-1650ms/call — a real but BOUNDED ~30% throughput penalty under concurrency (≈2.9-3.1x effective throughput from 4 workers, not the ideal 4x, but a genuine net win over any workload large enough to amortize it).
  3. The actual dominant bug, found by comparing end-to-end numbers against this per-call baseline: the "rolling pool, fresh worker per candidate" design from L61 attempt #3 paid a cold-JIT/module-load tax on EVERY candidate (a first-ever `RearrangeSolver`/`BoardSimulator` call in a brand-new isolate costs far more than a warmed-up one) — `solveDoraParallel`/`solveClearAllParallel` never hit this because they spawn K workers ONCE and feed each MANY items, amortizing warmup across a whole shard. Spawning fresh per candidate paid the full warmup tax on every single one.
  4. Fixed by switching to a PERSISTENT worker pool (K long-lived workers, bidirectional `postMessage` — master hands out one candidate at a time, worker replies and gets the next) so JIT warmup amortizes across a call. This alone still under-performed on the "easy, first candidate already succeeds" case, because ANY pool with K>1 workers unavoidably pays for K candidates running CONCURRENTLY before any of them can report back — there's no way to know in advance which will succeed. Sequential's lucky 1-candidate exit can't be beaten by a pool that always commits to K up front.
  5. Final fix: a small in-process SEQUENTIAL prefix (`min(4, candidates.length)`) tried BEFORE spinning up any workers at all — cheap insurance against the (empirically common, on the test boards used) case where an early candidate already satisfies the demand. Only once the prefix is exhausted without success does the persistent pool fan out across the REMAINING candidates — which is exactly the case (genuinely hard, every candidate needed) where the pool's real per-candidate throughput advantage actually pays for itself.
- Measured live, both directions: EASY (early-success) boards now run at ~parity with sequential (6952ms seq vs 7483ms parallel, ~8% overhead from the prefix's own worst-case, not the old design's 4x-worse blowup) — no regression. - Lesson: when a user asks "is this really impossible?" after a negative result, that's a legitimate prompt to re-derive from first principles rather than defend the prior conclusion — the earlier investigation (L61) wasn't WRONG about its own measurements, but it drew too broad a conclusion ("doesn't parallelize on this machine") from a narrower actual cause (specific design flaws: no fair control, fresh-worker-per-item cold-start, and no protection for the common early-success case). - Evidence: fair `solveClearAllParallel` control (1950ms seq vs 1003ms workers=4); pure-CPU busy-loop scaling (3038ms solo vs 3047ms at n=8); per-call warm-state timing inside persistent workers (solo ~1200-1300ms/call vs 4-way concurrent ~1550-1650ms/call — the real, bounded contention figure); final shipped design: EASY 6952ms seq vs 7483ms par4 (parity), HARD 171648ms seq vs 85178ms par4 vs 61612ms par8 (2.0x/2.8x), all replay-ok:true.

## L63 2026-07-11 `--screenshot-after-spin` captured mid-dissolve because it fired after the drag function's own post-release cleanup, not after the actual release

## L64 2026-07-11 L63's fix (capture right after release) still wasn't fast enough — the game's dissolve outraces an adb screencap round-trip; must capture BEFORE releasing, while still held
- Context: immediately after shipping L63's fix, User tested again live and reported the screenshot still showed the dissolve in progress — this time "4 Combo!!" (a LATER combo than L63's report of "3 Combo!!"), even though the capture now fired right after sending `u 0`.

- Lesson: when two different approaches to "capture the moment right after an action" both lose to how fast the action's real-world consequence unfolds, the fix isn't to shave the delay further — it's to capture from a moment where the consequence PROVABLY hasn't started yet (before the triggering action, not after it). This project already had that exact pattern solved for a different feature (`--after-spin-kill`'s mid-hold shot); the right move when adding a similar screenshot-timing feature is to check for an existing analogous case FIRST, not discover the same race condition from scratch via two rounds of live user reports.

## L65 2026-07-12 capturePng's two-round-trip screencap+pull was the real cost of the hold — switched to a single adb exec-out call
- Context: after L64's fix (capture while still held, before releasing), User asked directly whether the screenshot itself takes time to capture — noticing the hold felt "quite long."
- Root cause: `capturePng` (shared by `--screenshot-after-spin`'s new hold-time shot AND the pre-existing `--after-spin-kill` mid-hold/post-continue shots) ran TWO sequential adb invocations: `adb shell screencap -p /sdcard/tos_screen.png` (device encodes and writes a PNG to storage), then a completely separate `adb pull /sdcard/tos_screen.png <local>` (a second adb process, transferring the file back). Each `execSync` call blocks until that whole adb process exits — commonly several hundred ms to over a second combined, on top of the 300ms settle sleep already added by L64.
- Fix: replaced both calls with a single `adb exec-out screencap -p`, which streams the PNG directly to stdout in one adb invocation — no on-device file, no separate pull. `exec-out` (unlike plain `adb shell`, which allocates a PTY) doesn't corrupt binary output with newline translation, so no `encoding` option is needed on `execSync` — it returns a raw `Buffer`, written straight to `outFile`. Raised `maxBuffer` to 64MB (default 1MB can be smaller than an actual screenshot PNG). This is a shared helper, so the speedup applies to every screenshot in the codebase (`--after-spin-kill`'s two shots too), not just the new flag.
- Lesson: when a User asks "why does X take so long" about something built recently, take it as a genuine profiling prompt, not just reassurance-seeking — look at what the helper function ACTUALLY does under the hood (here: two full adb process spawns) rather than assuming the already-diagnosed cause (L64's dissolve race) is the only cost left.
- Evidence: code-level fix, device offline this session (could not measure the before/after wall-clock delta live) — timing improvement is architecturally expected (one adb round-trip removed) but not yet directly measured; worth confirming actual ms saved on next live test.
- Affected files: phone/autospin.js (`capturePng` rewritten); PROJECT-FACTS P58 updated to describe this third round inline.
